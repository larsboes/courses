Exercise 9: REST API & Docker

Transforming a local app into a client-server architecture

🎯 What We Built

We transformed the "Meta Playlist" app from a client-only application (storing data in your browser) into a full client-server web application.

🖥️ Backend

Python Flask server with REST API endpoints for managing playlists.

🌐 Frontend

JavaScript updated to use fetch() instead of localStorage.

🐳 Docker

Containerized for consistent deployment anywhere.

📋 OpenAPI

Formal API specification for documentation and tooling.

🤔 Why This Matters

Before (localStorage)

  • Data stored only in your browser
  • Switch browsers or devices? Data is gone.
  • Clear browser data? Everything deleted.
  • Want to share playlists? Not possible.

After (Server-side storage)

  • Data stored on a central server
  • Access from any device (if hosted publicly)
  • Data persists across browser sessions
  • Foundation for multi-user features
Note: Our current implementation uses in-memory storage, so data is lost when the server restarts. In production, you'd connect to a database like PostgreSQL or MongoDB.

🔌 REST API Explained

REST (Representational State Transfer) is an architectural style for designing APIs. Here's how we apply it:

Core Principles

Principle Our Implementation
Resources Playlists are our resources, identified by name
HTTP Methods GET (read), POST (create), PUT (update), DELETE (remove)
Stateless Each request contains all info needed; server doesn't track sessions
JSON Format Data exchanged as JSON objects

Our Endpoints

Method Endpoint Purpose
GET /api/playlists List all playlists
POST /api/playlists Create a new playlist
GET /api/playlists/{name} Get specific playlist
PUT /api/playlists/{name} Update playlist tracks
DELETE /api/playlists/{name} Delete playlist

📋 OpenAPI Specification

We created an openapi.yaml file that formally describes our API. This is the industry standard (formerly known as Swagger).

Why Use OpenAPI?

  • Documentation: Auto-generate interactive API docs (Swagger UI)
  • Code Generation: Generate client SDKs in any language
  • Validation: Automatically validate requests/responses
  • Testing: Generate test cases from spec
  • Collaboration: Contract between frontend and backend teams

Example from our spec:

paths:
  /playlists:
    post:
      summary: Create a new playlist
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required:
                - name
              properties:
                name:
                  type: string

🐳 Docker Explained

Docker packages your application with everything it needs to run into a container.

Our Dockerfile Breakdown

# Start with Python 3.9 (slim = minimal OS)
FROM python:3.9-slim

# Set working directory inside container
WORKDIR /app

# Install Flask framework
RUN pip install flask

# Copy our code into the container
COPY app.py .
COPY static ./static
COPY openapi.yaml .

# Tell Docker which port our app uses
EXPOSE 5000

# Command to run when container starts
CMD ["python", "app.py"]

Benefits

🔒 Isolation

App runs separately from your system

📦 Portability

Same container runs on Mac, Linux, Windows, Cloud

🔄 Reproducibility

"Works on my machine" is now everyone's machine

📐 Scalability

Easy to run multiple instances

Data Storage

Unlike the previous version which used volatile in-memory storage, this version persists data in a CouchDB database.

When you create or update a playlist, the Flask application sends a request to the CouchDB container to save the document. This ensures that your playlists survive container restarts.

🚧 What's Missing / Next Steps

Our current implementation is a prototype. Here's what a production system would need:

1. Persistent Storage (Database)

Current: Data stored in Python dictionary → Lost on restart
Solution: Add PostgreSQL, MongoDB, or SQLite 
# Example with SQLAlchemy ORM
from flask_sqlalchemy import SQLAlchemy
db = SQLAlchemy(app)

class Playlist(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    name = db.Column(db.String(100), unique=True)

2. Authentication & Authorization

  • Currently: Anyone can access any playlist
  • Need: User accounts, login, permissions
  • Options: JWT tokens, OAuth 2.0, session cookies

3. Input Validation

  • Validate playlist names (no special chars, length limits)
  • Validate track URLs (real streaming service links)
  • Sanitize input to prevent injection attacks

4. Error Handling

  • Proper HTTP status codes (400, 401, 404, 500)
  • Consistent error response format
  • Logging for debugging

5. Streaming Service Integration

  • Spotify API: Import playlists, fetch metadata
  • YouTube Music API: Link to actual tracks
  • Cross-platform playlist sync

6. Frontend Improvements

  • Use a framework (React, Vue, Svelte)
  • Add loading states and error messages
  • Implement search and filtering

7. DevOps & Production

  • HTTPS encryption (TLS certificates)
  • Environment variables for config
  • Health checks and monitoring
  • CI/CD pipeline for automated deployment

🧪 How to Test the API

You can use curl, Postman, or your browser's dev tools:

Create a Playlist

curl -X POST http://localhost:5001/api/playlists \
  -H "Content-Type: application/json" \
  -d '{"name": "My Rock Playlist"}'

Add Tracks

curl -X PUT http://localhost:5001/api/playlists/My%20Rock%20Playlist \
  -H "Content-Type: application/json" \
  -d '[
    {"title": "Bohemian Rhapsody", "link": "https://spotify.com/...", "duration": "5:55"},
    {"title": "Stairway to Heaven", "link": "https://spotify.com/...", "duration": "8:02"}
  ]'

View All Playlists

curl http://localhost:5001/api/playlists
